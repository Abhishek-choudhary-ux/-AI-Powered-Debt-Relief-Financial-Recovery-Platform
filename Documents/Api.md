# API Reference

Base URL (default): `http://127.0.0.1:8000`

The backend exposes **two parallel sets of endpoints**:

1. **Versioned API** (`/api/v1/...`) — the "real" implementation, routers in `Backend/app/api/*.py`, fully wired to the database and financial engine.
2. **Alias endpoints** (`/api/...`, no version prefix) — defined directly inside `Backend/app/main.py`, built specifically to match the frontend's exact expected paths and JSON field names. These are what the current React frontend actually calls.

Both are documented below since both are live and callable. Interactive docs are also auto-generated by FastAPI at `/docs` (Swagger) and `/redoc`.

---

## Versioned API (`/api/v1`)

### Health

#### `GET /api/v1/health/`
Returns service status.

**Response 200**
```json
{
  "status": "ok",
  "app": "AI Powered Debt Relief & Financial Recovery Platform",
  "environment": "development",
  "version": "v1"
}
```

### Authentication (`/api/v1/auth`)

#### `POST /api/v1/auth/register`
**Body**
```json
{ "name": "string (2-100 chars)", "email": "valid email", "password": "string (min 6 chars)" }
```
**Response 200**
```json
{ "success": true, "message": "Account created successfully", "token": "<JWT>", "user": {"id": 1, "name": "...", "email": "..."} }
```
**Errors**: `400` if email already registered.

#### `POST /api/v1/auth/login`
**Body**: `{ "email": "...", "password": "..." }`
**Response 200**: same shape as register.
**Errors**: `401` if credentials invalid.

#### `GET /api/v1/auth/me`
**Headers**: `Authorization: Bearer <token>`
**Response 200**: `{ "success": true, "user": {"id":1,"name":"...","email":"..."} }`
**Errors**: `401` if token invalid/expired or user not found.

> Passwords are hashed with `sha256_crypt` (via `passlib`), not bcrypt, despite `passlib[bcrypt]` being listed in `requirements.txt` — see [Security.md](Security.md).

### Loans (`/api/v1/loans`)

#### `POST /api/v1/loans/`
**Body** (`schemas.LoanCreate`):
```json
{
  "user_id": 1,
  "lender_name": "string",
  "loan_type": "string",
  "loan_amount": 100000.0,
  "interest_rate": 12.5,
  "outstanding_balance": 80000.0,
  "tenure_months": 24,
  "emi_due_date": "2025-01-15",
  "start_date": "2024-01-15"
}
```
**Response 200**: `{ "success": true, "loan_id": 5, "message": "Loan created successfully" }`

#### `GET /api/v1/loans/user/{user_id}`
Returns all loans for a user: `{ "user_id": 1, "loans": [...], "total": 3 }`

#### `GET /api/v1/loans/{loan_id}`
Returns a single loan object or `404`.

#### `DELETE /api/v1/loans/{loan_id}`
Deletes the loan. Returns `404` if not found.

> **Note**: there is no `PUT`/`PATCH` endpoint to update a loan's outstanding balance or status via the API layer, even though `DataBase/loan_settlement_service.record_payment()` implements that logic — it's simply never called from any router.

### Financial Analysis (`/api/v1/financial`)

#### `POST /api/v1/financial/analyze`
Stateless — no database write. **Body** (`FinancialAnalysisRequest`):
```json
{
  "loan_amount": 100000,
  "outstanding_balance": 80000,
  "interest_rate": 12.5,
  "tenure_months": 24,
  "monthly_income": 40000,
  "monthly_expenses": 25000,
  "emi_amount": null,
  "emi_due_date": null
}
```
**Response 200**: `{ "success": true, "analysis": { ...see Database_Schema.md / AI_Model.md for field meanings... } }`

#### `POST /api/v1/financial/monthly`
Saves a monthly income/expense record.
**Body**: `{ "user_id": 1, "month": "2025-01", "monthly_income": 40000, "monthly_expenses": 25000 }`
**Response 200**: `{ "success": true, "record_id": 7, "message": "Monthly financial data saved" }`

#### `GET /api/v1/financial/monthly/{user_id}`
Returns all monthly records for a user.

#### `POST /api/v1/financial/predict-settlement/{user_id}?monthly_income=...&monthly_expenses=...`
Uses the **alternate** scoring engine (`DataBase/settlement_prediction.py`), not `financial_engine.py`. Requires the user to already have active loans in the database. Returns per-loan settlement predictions plus a saved `FinancialMetric` row.
**Errors**: `400` if no active loans found or inputs invalid.

> This endpoint uses `monthly_income`/`monthly_expenses` as **query parameters**, not a JSON body — inconsistent with every other POST endpoint in the API, which use JSON bodies. This is a minor API design inconsistency worth normalizing.

### Settlement (`/api/v1/settlement`)

#### `POST /api/v1/settlement/recommend`
The full Scenario 1 pipeline: financial analysis → Gemini/fallback → DB save.
**Body**:
```json
{
  "user_id": 1, "loan_id": 5,
  "loan_amount": 100000, "outstanding_balance": 80000,
  "interest_rate": 12.5, "tenure_months": 24,
  "monthly_income": 40000, "monthly_expenses": 25000,
  "emi_amount": null, "notes": null
}
```
**Response 200**:
```json
{
  "success": true,
  "recommendation": {
    "source": "gemini | fallback",
    "recommendation_summary": "...",
    "negotiation_strategy": "...",
    "negotiation_letter": "...",
    "financial_tips": ["...", "...", "..."],
    "suggested_offer": 52000.0,
    "offer_percentage": 65.0,
    "health_score": 58.4,
    "stress_level": "MEDIUM",
    "settlement_id": 12,
    "analysis": { "...full financial_engine output..." }
  }
}
```
If the DB save fails, `settlement_id` is `null` and a `db_warning` field is added — the recommendation itself is still returned (partial-failure tolerant).

#### `GET /api/v1/settlement/user/{user_id}`
Returns all settlement records across all of a user's loans (joins via `loan.id`).

#### `GET /api/v1/settlement/{settlement_id}`
Single settlement record, or `404`.

  "interest_rate": 12.5, "tenure_months": 24,

  "success": true, "source": "gemini | fallback",

```json

```

## Alias endpoints (unversioned, defined in `main.py`)

These exist purely for frontend compatibility (see [Troubleshooting.md](Troubleshooting.md)) and, except where noted, **do not** persist or read real data.

| Method & Path | Backed by real logic? | Behavior |
|---|---|---|
| `POST /api/auth/register` | ✅ Yes | Delegates directly to `user.register()` |
| `POST /api/auth/login` | ✅ Yes | Delegates directly to `user.login()` |
| `GET /api/dashboard` | ❌ No | Always returns zeroed static JSON |
| `POST /api/settlement/predict` | ✅ Partially | Runs the real `financial_engine` + Gemini/fallback, but doesn't require/use `user_id`/`loan_id`, so nothing is persisted to a specific loan |
| `POST /api/negotiation/generate` | ✅ Partially | Same as above, tailored field names for the negotiation UI; nothing persisted |
| `GET /api/rights` | ❌ No (static content) | Hard-coded list of 6 rights topics |
| `GET /api/history` | ❌ No | Always returns an empty list |
| `GET /` | ✅ Yes | Simple welcome/info payload |

**Full field-name mapping accepted by `/api/settlement/predict`** (first non-null wins):

| Standard name | Alternate accepted name(s) |
|---|---|
| `loan_amount` | `original_balance` |
| `outstanding_balance` | `current_balance` |
| `tenure_months` | `months_delinquent` |
| `monthly_income` | `monthly_payment_capacity` |
| `monthly_expenses` | *(computed as 60% of income if absent)* |

**Full field-name mapping accepted by `/api/negotiation/generate`**:

| Standard name | Alternate accepted name(s) |
|---|---|
| `outstanding_balance` | `current_balance` |
| `offer_amount` | *(defaults to 50% of outstanding balance)* |
| `monthly_income` | `monthly_payment_capacity` (or `offer_amount * 3`) |
| `lender_name` | `creditor_name` |

## Error format

All errors follow FastAPI's default shape:
```json
{ "detail": "human-readable message" }
```
Custom exception classes (`app/utils/exceptions.py`) map to these status codes:

| Exception | HTTP Status |
|---|---|
| `NotFoundError` | 404 |
| `ValidationError` | 422 |
| `DatabaseError` | 500 |
| `AIServiceError` | 503 (defined but never actually raised anywhere — Gemini failures are always absorbed by the fallback instead) |
| `InsufficientDataError` | 400 (defined but never raised anywhere in the codebase) |

Pydantic validation failures (bad types, missing required fields, out-of-range values) are returned automatically by FastAPI as `422` with a `detail` array describing each failing field.
